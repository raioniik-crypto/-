## 🚀 Phase 1 でやること（実装タスク）

### 最終ゴール

**任意の GitHub リポジトリを対象にした `code_review` Routine を1件完走させる**

### 実装タスク（9項目）

#### 1. Supabase `job_locks` テーブル作成 SQL の実行
- Claude Code が SQL を生成
- いさむが Supabase SQL Editor に貼って Run
- 作成対象:
  - `job_locks` テーブル（TTL、scope、routine_name 等のカラム付き）
  - `acquire_lock` 関数（TTL 付き）
  - `release_lock` 関数
  - `cleanup_expired_locks` 関数
  - `check_lock` 関数（デバッグ用）

#### 2. Supabase 接続情報の環境変数設定
- 変数名:
  - `SUPABASE_URL`
  - `SUPABASE_ANON_KEY`
  - `SUPABASE_SERVICE_ROLE_KEY`
- 配置: Render ダッシュボードの新規 Worker 環境変数
- 注意: これはいさむが手動で設定

#### 3. Docker イメージの設計・作成
- ベース: `node:22-slim`（推奨）
- インストール: Claude Code CLI, git, OpenSSH
- ファイル: `voice-pipeline/Dockerfile.routine-worker`
- ビルドテスト: ローカルでビルド→起動確認

#### 4. `src/routine-worker.ts` の新規実装
- 主要機能:
  - Supabase から job を取得
  - `job_locks` で排他制御（`acquire_lock` 関数呼び出し）
  - Claude Code CLI を `--non-interactive` モードで起動
  - STDOUT から `[BLOCKER_DETECTED]` パターン検知
  - 結果を job ledger (GitHub Contents API) に反映
  - Vault に成果物を保存

#### 5. API に Routine エンドポイント追加
- `POST /routines`: Routine 起動（job 作成 + キュー投入）
- `GET /routines/:id`: Routine 状態取得
- `POST /routines/:id/retry`: 失敗時再試行
- **重要**: 既存 `/jobs/*` エンドポイントは一切変更しない

#### 6. Discord Bot にコマンド追加
- `/routine code_review repo:<url> target:<...>`
- 既存 8 コマンド（`/memo`, `/job`, etc）は無変更

#### 7. Render への新規 Worker デプロイ
- サービス名: `voice-pipeline-routine-worker`
- プラン: Standard ($25/月)
- Docker イメージ利用
- 環境変数設定

#### 8. `code_review` Routine の動作確認
- `rules/code_review.md` のテスト 1〜3 を実行
- 成果物が Vault に保存されることを確認
- Discord に完了通知が届くことを確認

#### 9. 朝サマリー自動生成スクリプト
- `templates/morning_summary_template.md` に従って生成
- Discord 配信（短縮版）+ Vault 保存（完全版）

## 🚨 Phase 1 で特に注意すべき点

### 重要度 S（必ず守る）

1. **既存 worker のコードを変更しない**
   - `src/worker.ts`, `src/jobs.ts`, `src/server.ts` の既存部分は触らない
   - Routine 関連は新規ファイル `src/routine-worker.ts` で完結させる
   - API にエンドポイント追加する際は新規ハンドラとして追加、既存は触らない

2. **禁止ファイルへの接触禁止**
   - `.env`, 認証ファイル, 秘密鍵
   - `rules/_common.md` の F1 を参照

3. **main ブランチへの直接 push 禁止**
   - すべての変更は `claude/` プレフィックスのブランチ経由
   - PR 経由でのマージは Phase 3 まで実装しない、Phase 1 ではブランチ push まで

### 重要度 A（強く推奨）

4. **小さい単位でコミット**
   - 論理的な単位（「Dockerfile 追加」「routine-worker スケルトン」等）で分割
   - 1コミットあたり 500 行以内推奨

5. **動作確認を段階的に**
   - ローカルで動作確認 → Render デプロイ → 本番動作確認の順

6. **ブロッカー時は即座に停止**
   - `rules/_common.md` の B1〜B7 に該当したら `[BLOCKER_DETECTED]` を出す
   - いさむ経由で Claude（会話スレ）に判断を仰ぐ
